We have yggstack app as an example. It's TUN-less implementation of yggdrasil-go used for port forwarding and proxy.
We have yggdns64 app as an example dns64 for yggdrasil.

We need to implement new app that will spin up TUN-less yggdrasil node and expose 2 services into yggdrasil nentwork: nat64 for the traffic from other yggdrasil nodes and yggdns64.

- single binnary file
- no TUN interface
- fully in userspace (no ROOT requred)
- restricted sources list for both services nat64 and dns64 (only particular subnets are allowed)

Some details:
Yggdrasil instance has 300/ subnet additionally for the main ip address. So we could use that subnet for nat64 pool. for example:
```
"pool6": "303:fafa:fefe:dede:ff::/96"
```
In this case if some other yggdrasil host will try to ping, it will work like this:
```
ping 303:fafa:fefe:dede:ff::1.1.1.1

Pinging 303:fafa:fefe:dede:ff:0:101:101 with 32 bytes of data:
Reply from 303:fafa:fefe:dede:ff:0:101:101: time=40ms
Reply from 303:fafa:fefe:dede:ff:0:101:101: time=41ms
Reply from 303:fafa:fefe:dede:ff:0:101:101: time=41ms
Reply from 303:fafa:fefe:dede:ff:0:101:101: time=41ms
```

Our app will start yggdrasil instance (without tun), will join the yggdrasil network using peers in the config and will provide yggdns64 and nat64 for the allowed sources.

We will extend yggdrasil config file with additional parameters. If you will run `ydn64 -genconf` you will receive default yggdrasil config (like for `yggdrasil -genconf`) with few additional parameters.

1) Pool6: "303:fafa:fefe:dede:ff::/96" - parameter that will show the subnet for nat64 logic.
2) AllowedSources: [] - the list of yggdrasil ipv6 addresses with prefix that will be allowed to pass traffic to nat64. It could be single ipv6 address or subnets in the list.

We will pre-generate Pool6 subnet for the config. We know our private key on config generation step. So, we understand our IP address and 300/ subnet.
